## 1. 제품 개요

### 1.1 한 줄 정의

**Learning-Clinic**은 교육기관 정보와 멘티 자가정보를 함께 분석해 멘티의 **역량 건강검진**을 만들고, 멘토가 AI의 도움을 받아 **맞춤형 멘토링 제안서**를 작성한 뒤, 멘티의 수락을 거쳐 멘토링을 시작하는 **멘토 전용 플랫폼**이다.

### 1.2 제품 철학

이 서비스의 핵심은 단순 매칭이 아니다.

> **멘토도 멘티를 잘 알고, 멘티도 멘토를 잘 알고 시작한다.**

기존 멘토링은 "일단 만나고 나서 서로 파악하는 방식"이었다면, Learning-Clinic은 **시작 전에 서로를 충분히 이해하고 합의한 뒤 시작하는 방식**을 만든다.

### 1.3 문제 정의

현재 멘토링은 다음 문제를 자주 겪는다.

- 멘토가 멘티를 충분히 이해하지 못한 상태에서 시작한다.
- 기관이 가진 정보와 멘티가 느끼는 실제 어려움이 연결되지 않는다.
- 멘토링 목표와 방식이 사전에 명확히 합의되지 않는다.
- 멘티는 "왜 이 멘토와, 왜 이 방식으로 시작하는지" 잘 모른 채 들어간다.
- 초반 핏 미스매치로 신뢰 형성이 늦고, 중도 이탈 가능성이 높아진다.

### 1.4 해결 방식

Learning-Clinic은 아래 흐름을 만든다.

```
기관 정보 수집 + 멘티 자가진단 → 역량 건강검진 → 멘토 AI 보조 계획 수립 → 멘티에게 제안 → 수락/조정 → 멘토링 시작
```

## 2. 목표와 비목표

### 2.1 목표

- 멘토가 멘티를 더 깊고 빠르게 이해할 수 있게 한다.
- 멘티가 멘토의 스타일과 제안을 충분히 이해한 뒤 시작하게 한다.
- 기관 정보와 멘티 자가정보를 결합한 정밀 인테이크 구조를 만든다.
- AI를 활용해 멘토의 계획 수립 시간을 줄이고 품질을 표준화한다.
- 멘토링 시작 전 단계의 신뢰 형성과 합의를 제품화한다.

### 2.2 비목표

- LMS 전체 기능을 대체하는 것
- AI가 멘토를 완전히 대신하는 것
- 심리 치료나 의료 진단을 수행하는 것
- 화상수업, 시험감독, 강의 제공 플랫폼이 되는 것
- 시작부터 대규모 공개 마켓플레이스형 매칭 플랫폼이 되는 것

## 3. 핵심 사용자

### 3.1 멘토

**역할:**
- 멘티 정보를 검토하고 역량 건강검진을 확정
- AI 도움을 받아 멘토링 계획과 제안서 작성
- 멘티와 조건을 합의한 뒤 멘토링 시작
- 멘토링 진행 및 기록 관리

**핵심 니즈:**
- 시작 전에 멘티를 잘 알고 싶다.
- 멘토링을 감이 아니라 구조적으로 설계하고 싶다.
- 나의 멘토링 스타일과 기대 역할을 명확히 전달하고 싶다.

### 3.2 멘티

**역할:**
- 자신의 상태, 목표, 어려움, 선호 방식을 제공
- 멘토의 스타일과 제안을 이해
- 수락/수정 요청/거절을 통해 멘토링 시작 여부 결정
- 수락 후 멘토링에 참여

**핵심 니즈:**
- 나를 제대로 이해한 멘토와 시작하고 싶다.
- 내가 왜 이 제안을 받는지 알고 싶다.
- 멘토의 방식이 나와 맞는지 시작 전에 알고 싶다.

### 3.3 교육기관 담당자

**역할:**
- 멘티 관련 기초 정보와 학습/평가 데이터를 전달
- 멘토링 의뢰 및 레퍼럴 생성
- 필요 시 진행 요약을 확인

**핵심 니즈:**
- 멘토에게 정확한 배경 정보를 넘기고 싶다.
- 멘토링이 어떤 기준으로 시작되는지 알고 싶다.
- 개입이 필요한 학생을 구조적으로 연결하고 싶다.

### 3.4 운영 관리자

**역할:**
- 기관, 멘토, 멘티 권한 관리
- 데이터 접근 정책 및 운영 현황 관리
- 품질/전환 지표 모니터링

| 대상 | 가치 제안 |
|------|-----------|

| **교육기관** | "기관 정보가 단순 전달에 그치지 않고, 실제 맞춤형 개입으로 연결된다." |

## 5. 제품 원칙

### 양방향 이해
멘토는 멘티를 이해하고, 멘티는 멘토를 이해해야 한다.

### 투명한 시작
어떤 정보가 활용되었고 어떤 계획이 제안되는지 명확해야 한다.

### 수락 기반 시작
멘토링은 배정으로 시작하지 않고 동의로 시작한다.

### AI는 보조자
AI는 초안 생성과 구조화에 기여하되, 최종 판단은 멘토가 한다.

### 설명 가능한 진단
건강검진 결과는 근거와 함께 제시되어야 한다.

## 6. 서비스 모델

### 6.1 정보 입력 채널

멘토가 멘티를 이해하기 위한 정보는 두 채널로 들어온다.

#### A. 교육기관 채널

기관이 제공하는 객관 정보:

- 기본 인적 사항
- 과정/기수/소속
- 출석, 과제, 평가 결과
- 담당자 코멘트
- 현재 수준 및 기관 관찰 포인트
- 의뢰 목적

#### B. 멘티 채널

멘티가 직접 제공하는 자기 정보:

- 목표
- 현재 어려움
- 자신감 수준
- 학습 습관
- 시간 여건
- 선호하는 멘토링 방식
- 피드백 선호도
- 기대하는 도움

### 6.2 핵심 산출물

이 제품의 핵심 결과물은 네 가지다.

1. **멘티 역량 건강검진 리포트**
2. **멘토 스타일 카드**
3. **멘토-멘티 핏 요약**
4. **멘토링 제안서**

### 6.3 시작 구조

```
건강검진 → 핏 확인 → 제안 → 수락 → 멘토링 시작
```

## 7. 주요 사용자 여정

### Step 1. 기관 의뢰 생성

기관 담당자가 멘티 정보를 등록하고 관련 자료를 업로드한다.

### Step 2. 멘티 온보딩

멘티가 자가진단 설문과 목표/상태 입력을 완료한다.

### Step 3. 멘토 정보 준비

멘토는 자신의 프로필, 멘토링 철학, 강점 분야, 피드백 스타일을 등록한다.

### Step 4. 역량 건강검진 생성

플랫폼이 기관 데이터와 멘티 데이터를 통합하여 AI 초안을 생성하고, 멘토가 검토/수정/확정한다.

### Step 5. 핏 요약 및 제안서 작성

멘토는 AI 초안을 바탕으로 이번 멘토링의 목표, 기간, 방식, 기대 역할을 정리한 제안서를 만든다.

### Step 6. 멘티 제안 확인

멘티는 다음 정보를 본다:

- 나의 건강검진 요약
- 이 멘토가 나를 어떻게 이해했는지
- 멘토의 스타일
- 제안된 멘토링 방식
- 나에게 기대하는 역할

### Step 7. 수락/수정 요청/거절

멘티가 상태를 선택한다:

- 수락
- 수정 요청
- 보류
- 거절

### Step 8. 멘토링 시작

수락되면 멘토링 워크스페이스가 자동 생성되고 일정/체크인/세션 관리가 시작된다.

## 8. 기능 요구사항

### 8.1 계정 및 권한 관리

**P0**

- 사용자 역할은 멘토, 멘티, 교육기관 담당자, 운영자로 구분한다.
- 멘토는 자신에게 배정된 케이스만 볼 수 있다.
- 멘티는 자신의 공개 정보와 제안서, 시작 이후 자신의 멘토링 정보만 볼 수 있다.
- 기관 담당자는 자신이 의뢰한 케이스의 기관 입력 정보와 허용된 진행 요약만 볼 수 있다.
- 운영자는 전체 시스템 접근 권한을 가진다.

**P1**

- 기관별 권한 범위 커스터마이징
- 멘티 동의 여부에 따른 기관 열람 제한

### 8.2 기관 의뢰 입력

**P0**

- 기관 담당자는 멘티 의뢰 케이스를 생성할 수 있어야 한다.
- 입력 필드는 최소 아래를 포함한다:
  - 멘티 기본 정보
  - 소속 과정
  - 의뢰 목적
  - 기관이 보는 현재 상태
  - 과제/평가/출석 등 핵심 데이터
  - 참고 코멘트
- 파일 첨부 또는 폼 입력이 가능해야 한다.

**P1**

- CSV 업로드 지원
- 외부 LMS/API 연동

### 8.3 멘티 온보딩 및 자가진단

**P0**

- 멘티는 온보딩 링크를 통해 자가진단을 제출할 수 있어야 한다.
- 설문은 최소 아래를 포함한다:
  - 목표
  - 현재 고민
  - 자신감
  - 학습 습관
  - 시간/환경 제약
  - 선호 멘토링 방식
  - 원하는 피드백 강도
  - 기대하는 결과
- 멘티는 자신의 정보가 멘토링 제안에 활용된다는 점을 확인해야 한다.

**P1**

- 설문 템플릿 기관별 커스터마이징
- 재온보딩/업데이트 제출

### 8.4 멘토 프로필 및 스타일 카드

**P0**

- 멘토는 자신의 프로필을 등록할 수 있어야 한다.
- 프로필은 최소 아래를 포함한다:
  - 소개
  - 전문 분야
  - 잘 돕는 유형
  - 멘토링 철학
  - 피드백 스타일
  - 진행 방식
  - 선호 커뮤니케이션 방식
- 멘티는 제안서 확인 단계에서 멘토 프로필을 볼 수 있어야 한다.

**P1**

- 짧은 소개 영상
- 멘토 후기/평판 요소

### 8.5 역량 건강검진 리포트

**P0**

- 시스템은 기관 정보와 멘티 자가정보를 결합해 AI 기반 초안을 생성해야 한다.
- 멘토는 초안을 수정/확정할 수 있어야 한다.
- 건강검진 리포트는 다음을 포함해야 한다:
  - 현재 상태 요약
  - 강점
  - 위험 신호
  - 목표 대비 현재 위치
  - 역량 축별 평가
  - 기관 관찰 vs 멘티 자기인식 차이
  - 권장 개입 포인트
- 멘토 확정 전에는 멘티에게 공유되지 않는다.

**권장 역량 축:**

- 목표 명확성
- 기초 역량
- 실행력
- 지속 습관
- 자신감
- 피드백 수용성
- 커뮤니케이션
- 리스크 수준

**P1**

- 주차별/시점별 건강검진 누적 비교
- 기관별 커스텀 축 설정

### 8.6 멘토-멘티 핏 요약

**P0**

- 시스템은 멘토 스타일과 멘티 선호도를 비교한 핏 요약을 제공해야 한다.
- 핏 요약에는 아래가 포함되어야 한다:
  - 잘 맞는 요소
  - 조정이 필요한 요소
  - 진행 시 주의 포인트
- 멘토는 핏 요약을 검토 후 수정 가능해야 한다.

> **이 기능은 제품의 핵심 차별점이다.**
> 단순 프로필 열람이 아니라, 왜 이 멘토링 방식이 적합한지 설명하는 기능이어야 한다.

### 8.7 멘토링 제안서 작성

**P0**

- 멘토는 건강검진을 바탕으로 멘토링 제안서를 작성할 수 있어야 한다.
- AI는 제안서 초안을 생성할 수 있어야 한다.
- 제안서 필수 항목:
  - 제안 제목
  - 멘토링 목표
  - 제안 이유
  - 예상 기간
  - 주간 진행 방식
  - 멘토가 제공할 지원
  - 멘티에게 기대하는 역할
  - 첫 주 액션 아이템
  - 성공 기준
- 제안서 공유 전 멘토가 최종 검토해야 한다.

**P1**

- 제안서 템플릿 저장
- 유형별 제안서 라이브러리

### 8.8 멘티 검토 및 수락 워크플로우

**P0**

- 멘티는 제안서를 확인하고 아래 상태 중 하나를 선택할 수 있어야 한다:
  - 수락
  - 수정 요청
  - 보류
  - 거절
- 수정 요청 시 멘티는 사유를 입력할 수 있어야 한다.
- 수락 전에는 멘토링 워크스페이스가 생성되지 않는다.
- 수락 후에는 시작 일자와 첫 액션이 자동 설정된다.

**P1**

- 수락 시 전자 동의/약관 연결
- 보류 상태 리마인드

### 8.9 멘토링 시작 워크스페이스

**P0**

- 수락된 케이스는 멘토링 전용 워크스페이스로 전환되어야 한다.
- 워크스페이스는 아래를 포함해야 한다:
  - 목표
  - 일정
  - 주간 체크인
  - 세션 기록
  - 다음 액션
  - 진행 상태
- 멘토는 세션 단위 기록을 남길 수 있어야 한다.
- 멘티는 자신에게 공개된 액션과 피드백을 확인할 수 있어야 한다.

**P1**

- 리마인더
- 반복 체크인 자동화
- 세션 회고 템플릿

### 8.10 커뮤니케이션

**P0**

- 멘토와 멘티는 제안 단계에서 메시지를 주고받을 수 있어야 한다.
- 메시지는 케이스 맥락 안에 저장되어야 한다.
- 수정 요청, 질문, 확인 메시지를 구분할 수 있어야 한다.

**P1**

- 기관 담당자와의 삼자 메시지
- 음성/화상 연동

### 8.11 대시보드

#### 멘토 대시보드

**P0**

- 신규 의뢰
- 미완료 인테이크
- 건강검진 작성 대기
- 제안서 작성/응답 상태
- 진행 중 멘토링 목록

#### 멘티 대시보드

**P0**

- 자가진단 상태
- 받은 제안서
- 나의 멘토 정보
- 수락 후 진행 중 목표와 다음 액션

#### 기관 대시보드

**P0**

- 의뢰 케이스 목록
- 멘티 응답 여부
- 제안 전환 상태
- 시작 여부

## 9. AI 요구사항

### 9.1 AI 역할

AI는 아래 업무를 지원한다:

- 기관/멘티 정보 통합 요약
- 건강검진 초안 생성
- 핏 요약 초안 생성
- 멘토링 제안서 초안 생성
- 멘토링 목표와 액션 아이템 추천

### 9.2 AI 입력

- 기관 의뢰 데이터
- 멘티 자가진단 데이터
- 멘토 프로필/스타일
- 과거 유사 템플릿 또는 멘토가 저장한 문서

### 9.3 AI 출력

- 건강검진 요약
- 강점/리스크 태그
- 기관 관찰과 자기인식 차이 설명
- 제안 목표 초안
- 진행 방식 제안
- 첫 주 액션 제안

### 9.4 AI 가드레일

- AI는 최종 진단을 자동 확정할 수 없다.
- AI는 의료/정신건강 진단 문구를 사용하지 않는다.
- AI는 근거 없는 단정 대신 "관찰 기반 추정"으로 표현해야 한다.
- 멘토는 AI 출력을 수정, 재생성, 거절할 수 있어야 한다.
- 멘티에게 공개되는 문구는 비낙인적이고 행동 중심이어야 한다.

## 13. 비기능 요구사항

### 13.1 사용성

- 멘토는 한 케이스 건강검진 초안을 10분 내 검토할 수 있어야 한다.
- 멘티는 5분 내 제안서 핵심을 이해할 수 있어야 한다.
- 기관 담당자는 3분 내 의뢰 생성이 가능해야 한다.

### 13.2 보안

- 역할 기반 접근 제어 필수
- 민감한 자기서술 정보는 최소 권한 원칙 적용
- 모든 문서 수정 및 공유 이력 기록

### 13.3 신뢰성

- AI 생성 버전과 멘토 최종 버전을 분리 저장
- 제안 상태 변경 이력 보존
- 수락/거절 시점 기록

### 13.4 성능

- 대시보드 3초 이내 로드 목표
- AI 초안 생성 15초 이내 목표

## 14. 수용 기준

### 14.1 건강검진

- 기관 입력과 멘티 자가진단이 완료되면 건강검진 생성이 가능해야 한다.
- 건강검진은 멘토 검토 후에만 확정 가능하다.
- 확정 전까지 멘티에게 노출되지 않는다.

### 14.2 제안서

- 제안서에는 목표, 방식, 기간, 기대 역할이 반드시 포함되어야 한다.
- 멘토는 공유 전 최종 검토를 완료해야 한다.
- 공유 후 상태는 응답 대기로 전환된다.

### 14.3 멘티 응답

- 멘티는 최소 수락, 수정 요청, 거절을 선택할 수 있어야 한다.
- 수정 요청 시 코멘트를 남길 수 있어야 한다.
- 수락 전에는 워크스페이스가 생성되지 않는다.

### 14.4 워크스페이스

- 수락 직후 멘토링 워크스페이스가 자동 생성되어야 한다.
- 최소 1개의 첫 액션 아이템이 포함되어야 한다.

## 19. 최종 요약

Learning-Clinic의 MVP는 **멘토링 자체보다 멘토링을 시작하는 방식**을 바꾸는 데 초점을 둔다.

### 기존 방식

```
배정 → 첫 만남 → 서로 파악
```

### Learning-Clinic 방식

```
정보 수집 → 역량 건강검진 → 멘토 스타일/핏 확인 → 맞춤 제안 → 멘티 수락 → 멘토링 시작
```

즉, 이 제품은 단순 멘토링 툴이 아니라,

> **상호 이해와 동의를 기반으로 멘토링을 설계하고 시작하게 만드는 멘토 중심 플랫폼**

이다.
